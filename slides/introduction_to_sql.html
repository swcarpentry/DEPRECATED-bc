Hello, and welcome to our section on SQL.

In this section, we're going to be talking about how you use and interact with your data.

Often, researchers might manually move data around, or save different calculations of the same data in different spreadsheets or in different places within the same spreadsheet.

In these cases, researchers are using spreadsheets, when they often really want to be using "databases."

What is Structured Query Language?

  • Is the primary language for "relational" databases
  • Allows searching ("querying") tables of data
  • Can help to automate searches and calculations often performed in spreadsheets
  • Easy to read and write

SQL stands for "Structured Query Languge."
[Click] It is a syntax for searching through and making calculations from spreadsheets of data (here called "tables").

[Click] This makes data more reusable, and your workflow more reproducible. Changes you make to data can all be written out, step-by-step.

[Click] Compared to many other languages, SQL is easy to read and write. It's also ubiquitous — It's usable with plugins for many languages, including R [sqldf] and Python [pandasql]. It even works with an add-on for Firefox [SQLite Manager plugin].

Let's say that you have data from a study of a new medication. You have two spreadsheets. First, a (shuffled) list of consent forms. These have each person's real name and contact info. Second, a spreadsheet of participation data. These are supposed to be anonymized and separated from the consent forms.

But what if you realize that everyone who's taken "Compound A" is likely to become very sick? Perhaps you need to identify everyone who has taken Compound A, and re-identify their data!

Affected Participants

ID Participation_Date Drug
2 2000-08-03 Compound A
4 2000-08-03 Compound A
5 2000-08-03 Compound A
9 2000-08-02 Compound A
10 2000-08-01 Compound A 
SELECT
    ID,
    Participation_Date,
    Drug
FROM participant_data
WHERE Drug = 'Compound A'

What you need, to start: a list of everyone who's taken Compound A.

A spreadsheet program like Excel could allow you to do this, but it might take several searches, manual copying and pasting, etc.

In SQL, it only takes 3 lines to filter / search through this list: SELECT the columns you want, from where, under what conditions. Easy to read, even at a glance!

SELECT
  participant_data.ID,		
  consent_forms.Name,
  consent_forms.Contact_Info,
  consent_forms.Participation_Date,
  participant_data.Drug,
  participant_data.Researcher_Initials
FROM consent_forms
JOIN participant_data
ON 
  consent_forms.Participation_Date = participant_data.Participation_Date
  AND consent_forms.Birthdate = participant_data.Birthdate

Eventually, you would want to link the Consent Form table with the Participant data table, allowing you to see the name of everyone who took Compound A.

As before, this could be done manually with a spreadsheet program. But look at how easy it is with SQL!

SELECT the columns you want, from where. JOIN (i.e., mash the second table onto the side of the first table) under certain extra conditions. Easy to read!
